🚀 **Software Architecture (Part 3): Abstract Factory Design Pattern**

Build families. Not just objects.

The Abstract Factory is a GoF Creational pattern that creates entire *families* of related objects — without coupling your code to their concrete types.


🔒 Intent

Provide an interface for creating families of related objects without specifying their concrete classes.


🌍 Real-World Scenario — Payment Processing

Your e-commerce app needs to support Stripe and PayPal. Each provider has its own way to create a payment session, handle a refund, and generate a receipt. They must always stay in sync — a Stripe session should never be paired with a PayPal refund.

Abstract Factory enforces that guarantee at the type level.


💻 C# Example


interface IPaymentSession { void Authorize(decimal amount); }
interface IRefundHandler  { void Refund(string transactionId); }

interface IPaymentFactory
{
    IPaymentSession CreateSession();
    IRefundHandler  CreateRefundHandler();
}

class StripeFactory : IPaymentFactory
{
    public IPaymentSession CreateSession()       => new StripeSession();
    public IRefundHandler  CreateRefundHandler() => new StripeRefund();
}

class PayPalFactory : IPaymentFactory
{
    public IPaymentSession CreateSession()       => new PayPalSession();
    public IRefundHandler  CreateRefundHandler() => new PayPalRefund();
}

// Checkout knows nothing about Stripe or PayPal
class CheckoutService(IPaymentFactory factory)
{
    public void Pay(decimal amount) => factory.CreateSession().Authorize(amount);
    public void Cancel(string id)   => factory.CreateRefundHandler().Refund(id);
}


Visual: IPaymentFactory → StripeFactory & PayPalFactory. Each produces a matched Session + RefundHandler pair. CheckoutService points only to IPaymentFactory — swap providers without changing a single line of checkout logic.


👍 Pros

✔ Consistency — products from the same factory are always compatible
✔ Open/Closed — add a new family without touching client code
✔ Testability — swap in a MockUIFactory with zero client changes


👎 Cons

✘ Complexity — many interfaces and classes for simple scenarios
✘ Rigidity — adding a new product type forces changes in every factory


💡 The Golden Rule

➖ Reach for Abstract Factory when you need to guarantee that a family of objects work together — multiple DB providers, UI themes, payment gateways, or notification channels.

If you only have one product type, Factory Method is enough.
